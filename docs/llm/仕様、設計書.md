## 3. ELECTRA Embedding 基礎

### ディレクトリ構成

```
study-electra-embedding/
├── data/
│   ├── corpus.jsonl          # 検索対象ドキュメント（20〜30件）
│   ├── queries.jsonl         # クエリ + relevant_docs（10件程度）
│   └── train_pairs.jsonl     # FT用ペアデータ
├── src/
│   ├── embed.py              # Embedding生成
│   ├── index.py              # FAISSインデックス構築
│   ├── search.py             # ベクトル検索
│   ├── train.py              # ファインチューニング
│   └── evaluate.py           # Recall@K / MRR算出
├── models/
│   ├── base/                 # ベースモデルキャッシュ
│   └── finetuned/            # FT済みモデル保存先
├── notebooks/
│   └── explore.ipynb
├── requirements.txt
└── Makefile
```

### 各スクリプトの責務

| スクリプト | 責務 |
|-----------|------|
| `embed.py` | テキスト→ELECTRAトークナイズ→[CLS]トークンのベクトル取得→numpy配列保存 |
| `index.py` | embed.pyの出力からFAISSインデックス構築→ファイル保存 |
| `search.py` | クエリテキストをEmbed→FAISSで近傍検��→Top-K結果返却 |
| `evaluate.py` | queries.jsonl読み込み→検索実行→relevant_docsと照合→Recall@K/MRR算出 |
| `train.py` | train_pairs.jsonlでContrastive Learning→FT済みモデル保存 |

### サンプルデータ設計

**corpus.jsonl（検索対象）**
- 日本語短文テキスト20〜30件
- ドメイン: FAQ系（商品説明・ニュース・FAQ等）

**queries.jsonl（検索クエリ + Ground Truth）**

```json
{
  "query_id": "q001",
  "query": "返品手続きの方法",
  "relevant_docs": ["d003", "d007", "d012"]
}
```

- 10件程度
- relevant_docsが正解ラベル（複数正解を許容）

## 4. 評価基盤

### 評価指標

| 指標 | 説明 | 例 |
|------|------|-----|
| Recall@K | 正解ドキュメントがTop-Kに何件含まれるか | 正解3件中2件がTop-5 → 0.67 |
| MRR | 最初の正解ドキュメントが何番目に出たか（逆数の平均） | 3番目に正解 → 1/3 = 0.33 |

### evaluate.py 処理フロー

```
1. queries.jsonlを読み込む
2. 各クエリでsearch.pyを実行（Top-K取得）
3. relevant_docsと照合
4. Recall@K / MRRを算出
5. 結果をJSONで出力
```

### 出力例

```json
{
  "recall@1": 0.40,
  "recall@3": 0.65,
  "recall@5": 0.80,
  "mrr": 0.52
}
```

## 5. ファインチューニング

### なぜEmbeddingのFTは特殊か

```
分類モデルのFT（一般的）
  └── [CLS] → Linear → クラス予測
  └── Cross Entropyで学習

Embedding用のFT
  └── [CLS] → ベクトル
  └── 類似ペアで学習させる必要がある
       └── Contrastive Learning
       └── または Sentence-BERT方式
```

### 学習データ設計

**train_pairs.jsonl**

```json
{
  "query": "返品手続きを教えてください",
  "positive": "商品の返品は購入後30日以内に手続きください",
  "negative": "新規会員登録の方法はこちらです"
}
```

- positive: 似ているべきテキスト（queryと近づける）
- negative: 似ていないべきテキスト（queryと遠ざける）

### train.py 処理フロー

```
1. train_pairs.jsonlを読み込む
2. ベースELECTRAをロード
3. Triplet Loss or Contrastive Lossで学習
   └── query と positive は近づける
   └── query と negative は遠ざける
4. models/finetuned/ に保存
```

### ベースモデル vs FTモデルの比較（Champion/Challenger）

```
ベースモデル評価結果
  recall@5: 0.60, mrr: 0.45

FTモデル評価結果
  recall@5: 0.75, mrr: 0.58

この比較体験が業務の
Champion/Challenger設計に直結する
```

精度向上幅は小さくてよい。「仕組みとして比較できる」ことが目的。

## 7. Vertex RAG MLOps

ELECTRA で Embedding 基礎を理解した上で、GCP上で実務レベルの RAG を MLOps として構築する。

### アーキテクチャ

```
[データ投入]
  ↓
GCS / 外部データ
  ↓
Cloud Run Job（前処理・Embedding生成）
  ↓
BigQuery（Vector Store）
  ↓
Cloud Run Service（API）
  ↓
Vertex AI（Gemini）
  ↓
レスポンス
```

### コンポーネント構成

| 層 | コンポーネント | 役割 |
|----|---------------|------|
| データ層 | GCS | 原データ格納 |
| データ層 | BigQuery | embedding格納・メタ��ータ・Vector Search |
| ML層 | Vertex AI Embedding API | Embedding生成 |
| ML層 | Vertex AI Gemini | 回答生成 |
| アプリ層 | Cloud Run Job | 前処理・embedding生成・BigQuery登録 |
| アプリ層 | Cloud Run Service | クエリ受付・類似検索・プロンプト生成・LLM呼び出し |

### データモデル（BigQuery）

```sql
table: documents

- id STRING
- content STRING
- embedding ARRAY<FLOAT64>
- created_at TIMESTAMP

-- インデックス: VECTOR INDEX（embedding）
```

### 処理フロー

**バッチ（インデックス生成）**

```
1. GCSからデータ取得
2. テキスト分割（chunking）
3. Vertex Embedding生成
4. BigQueryに格納
```

**リクエスト処理（RAG）**

```
1. ユーザークエリ受信
2. Embedding生成
3. BigQueryで類似検索（Top-K取得）
4. プロンプト生成（検索結果 + クエリ）
5. Geminiで回答生成
```

### 実装リファレンス

**Embedding生成**

```python
from vertexai.language_models import TextEmbeddingModel

model = TextEmbeddingModel.from_pretrained("textembedding-gecko")
emb = model.get_embeddings(["sample text"])
```

**類似検索（BigQuery）**

```sql
SELECT *
FROM documents
ORDER BY VECTOR_DISTANCE(embedding, @query_embedding)
LIMIT 5
```

**LLM呼び出し**

```python
from vertexai.generative_models import GenerativeModel

model = GenerativeModel("gemini-pro")
response = model.generate_content(prompt)
```

### Terraform構成

```
terraform/
├── main.tf
├── variables.tf
├── outputs.tf
├── cloud_run.tf      # Cloud Run Job + Service
├── bigquery.tf       # rag_dataset + documents テーブル

```

| リソース | 用途 |
|---------|------|
| `google_storage_bucket` | 原データ格納 |
| `google_bigquery_dataset` + `table` | embedding + メタデータ格納 |

| `google_cloud_run_service` | API（クエリ→検索→Gemini→レスポンス） |
| `google_service_account` + `iam_binding` | 最小権限 |
