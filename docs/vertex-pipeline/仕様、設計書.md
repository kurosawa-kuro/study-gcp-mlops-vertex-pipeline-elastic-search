# Vertex AI Pipeline 導入設計書
## study-gcp-mlops への Vertex AI Pipeline 統合

---

## 1. 目的

```text
既存の Cloud Run Job（力技スクリプト）を段階的に Vertex AI Pipeline に移行する

・RandomForest 学習後の評価・デプロイ判断を Pipeline として実装する
・品質ゲートを仕組みとして組み込む
・Champion/Challenger パターンを実装し、モデル切り替えを自動化する
・Vertex AI Pipeline の概念と実装を一致させる土台を作る
```

---

## 2. 既存構成との対応

### 現状（Cloud Run Job）

```text
Cloud Run Job（batch）1スクリプト
  ├── California Housing データ取得
  ├── train/test 分割
  ├── RandomForest 学習
  ├── 評価（RMSE, MAE）→ MLflow 記録
  ├── モデル保存 → GCS
  └── メトリクス投入 → BigQuery
```

### 目指す姿（Vertex AI Pipeline）

```text
Vertex AI Pipeline
  ├── Step1: データ取得・分割       ← Cloud Run Job から切り出し
  ├── Step2: 学習                  ← Cloud Run Job から切り出し
  ├── Step3: 評価（RMSE/MAE算出）  ← Cloud Run Job から切り出し
  ├── Step4: 品質ゲート            ← 新規追加（閾値チェック）
  ├── Step5: Champion/Challenger 比較 ← 新規追加
  └── Step6: デプロイ判断・実行    ← 新規追加
```

### 既存コンポーネントの扱い

| コンポーネント | 扱い | 理由 |
|---|---|---|
| BigQuery（metricsテーブル） | そのまま活用 | Champion モデル取得に流用 |
| GCS（models/） | そのまま活用 | モデルアーティファクト保存先 |
| Cloud Run Service（推論API） | そのまま | Pipeline の対象外 |
| Cloud Scheduler | Pipeline トリガーに置き換え | Vertex AI Pipeline 定期実行 |
| Discord 通知 | Pipeline 完了イベントに接続 | 通知ロジックは再利用 |
| MLflow | Vertex AI Experiments と併用 | 移行コストを考慮し段階的に |

## 3. 導入スコープ（フェーズ設計）

### 導入方針

```text
既存の Cloud Run Job（batch/api）は触らない（そのまま並行稼働）
既存の Python スクリプト（dataset.py, train.py 等）のロジックを
Vertex AI Pipeline のコンポーネントとして再実装する

理由：
・Cloud Run Job は既に動いており、壊すリスクを取らない
・同じロジックを Pipeline 化することで DAG・アーティファクト管理・品質ゲートを学べる
・段階的に Pipeline 側をメインに切り替えられる
```

### Phase 1（本設計書のスコープ）

```text
既存 Cloud Run Job は触らない（並行稼働）
既存スクリプトのロジックを Pipeline コンポーネントとして再実装し
データ取得 → 学習 → 評価 → 品質ゲート → Champion 比較 → デプロイを
一本の Vertex AI Pipeline として実装する

Pipeline への入力
  └── ハイパーパラメータ（n_estimators, max_depth 等）

Pipeline の処理
  ├── Step1: データ取得・分割       ← batch/dataset.py 相当
  ├── Step2: 学習                  ← batch/train.py 相当
  ├── Step3: 評価（RMSE/MAE算出）  ← batch/train.py evaluate 相当
  ├── Step4: 品質ゲート            ← 新規（閾値チェック）
  ├── Step5: Champion/Challenger 比較 ← 新規（BigQuery 連携）
  └── Step6: デプロイ判断・実行    ← 新規（Vertex AI Endpoint）

Pipeline の出力
  ├── GCS にモデル保存
  ├── BigQuery にメトリクス記録
  └── Vertex AI Endpoint へのデプロイ or スキップ
```

### Phase 2（将来）

```text
Cloud Run Job を廃止し Pipeline に完全移行
トラフィック分割（10% → 100%）段階的デプロイ
Vertex AI Experiments との深い統合
```

## 4. アーキテクチャ

### Pipeline 全体フロー

```text
[Cloud Run Job（既存・変更なし・並行稼働）]
  └── 従来通り動作（学習 → GCS保存 → BigQuery投入）

[Vertex AI Pipeline（新規・既存スクリプトのロジックを再実装）]
  │
  ├── Step1: load_data
  │     ├── California Housing データセット取得
  │     ├── train/test 分割
  │     └── データを Output Artifact として次 Step へ
  │         （← batch/dataset.py 相当）
  │
  ├── Step2: train_model
  │     ├── RandomForestRegressor 構築・学習
  │     ├── 学習済みモデルを Output Artifact として次 Step へ
  │     └── ハイパーパラメータを Vertex AI Experiments に記録
  │         （← batch/train.py build_model + train 相当）
  │
  ├── Step3: evaluate_model
  │     ├── テストデータでモデル評価
  │     ├── RMSE / MAE 算出
  │     ├── モデルを GCS に保存
  │     ├── メトリクスを BigQuery に記録
  │     └── Vertex AI Experiments にメトリクス記録
  │         （← batch/train.py evaluate + bq_store.py + model_store.py 相当）
  │
  ├── Step4: quality_gate
  │     ├── RMSE が閾値以下か判定
  │     ├── 合格 → Step5 へ進む
  │     └── 不合格 → Pipeline 終了 + Discord 通知
  │
  ├── Step5: compare_champion
  │     ├── BigQuery から Champion モデルのメトリクスを取得
  │     ├── Champion と Challenger の RMSE 比較
  │     ├── Challenger が優位（-X% 以上改善）→ Step6 へ
  │     └── 改善なし → Pipeline 終了 + Discord 通知
  │
  └── Step6: deploy_model
        ├── Vertex AI Endpoint にデプロイ
        ├── トラフィック切り替え
        └── BigQuery に Champion 更新 + Discord 通知

[Vertex AI Endpoint]
  └── Cloud Run Service（推論 API）から呼び出し
```

### 既存スクリプトとの対応表

```text
既存スクリプト                      → Pipeline コンポーネント
─────────────────────────────────────────────────────────
batch/dataset.py load_data()        → Step1: load_data
batch/train.py build_model/train()  → Step2: train_model
batch/train.py evaluate()           → Step3: evaluate_model
batch/model_store.py save_gcs()     → Step3: evaluate_model（内包）
batch/bq_store.py insert_metrics()  → Step3: evaluate_model（内包）
（新規）                            → Step4: quality_gate
（新規）                            → Step5: compare_champion
（新規）                            → Step6: deploy_model
```

## 6. 各 Step の詳細設計

### Step1: load_data

```python
# 責務
# - California Housing データセットを取得
# - train/test に分割
# - Output Artifact として次 Step にデータを渡す
# （← batch/dataset.py load_data() 相当）

# 入力（Pipeline パラメータ）
test_size: float      # デフォルト: 0.2
random_state: int     # デフォルト: 42

# 出力（Output Artifact）
X_train: Dataset      # 学習データ
X_test: Dataset       # テストデータ
y_train: Dataset      # 学習ラベル
y_test: Dataset       # テストラベル
```

### Step2: train_model

```python
# 責務
# - RandomForestRegressor を構築・学習
# - ハイパーパラメータを Vertex AI Experiments に記録
# - 学習済みモデルを Output Artifact として渡す
# （← batch/train.py build_model() + train() 相当）

# 入力
X_train: Dataset      # Step1 の出力
y_train: Dataset      # Step1 の出力
n_estimators: int     # デフォルト: 100
max_depth: int        # デフォルト: 10
random_state: int     # デフォルト: 42

# 出力（Output Artifact）
model: Model          # 学習済み RandomForestRegressor
```

### Step3: evaluate_model

```python
# 責務
# - テストデータでモデルを評価（RMSE / MAE 算出）
# - モデルを GCS に保存
# - メトリクスを BigQuery に記録
# - Vertex AI Experiments にメトリクスを記録
# - 次 Step にメトリクスを渡す
# （← batch/train.py evaluate() + model_store.py + bq_store.py 相当）

# 入力
model: Model          # Step2 の出力
X_test: Dataset       # Step1 の出力
y_test: Dataset       # Step1 の出力

# 出力
rmse: float
mae: float
model_gcs_path: str   # GCS 保存先パス
```

### Step4: quality_gate

```python
# 責務
# - RMSE が閾値以下かを判定
# - 不合格の場合は Pipeline を終了する
# - 閾値は Pipeline パラメータで外部化

# 入力
rmse: float           # Step3 の出力
rmse_threshold: float  # デフォルト: 0.8（調整可能）

# 出力
is_passed: bool

# 品質ゲート不合格時の動作
# - Pipeline を終了（後続 Step を実行しない）
# - Discord に通知（モデルパス・RMSE・閾値を含む）
```

### Step5: compare_champion

```python
# 責務
# - BigQuery から現 Champion モデルのメトリクスを取得
# - Challenger（新モデル）と RMSE を比較
# - 有意な改善がある場合のみ次 Step に進む

# 入力
challenger_rmse: float        # Step3 の出力
challenger_model_path: str    # Step3 の出力

# BigQuery クエリ（既存テーブル活用）
# SELECT model_path, rmse
# FROM mlops.metrics
# ORDER BY rmse ASC LIMIT 1

# 出力
should_deploy: bool
champion_rmse: float

# 比較基準（設計上の重要な決定事項）
# 単純比較ではなく「有意な改善」を定義する
# 例: challenger_rmse < champion_rmse * 0.99
#     （1%以上の改善がある場合のみデプロイ）
# 理由: モデルのランダム性によるブレを除外するため
```

### Step6: deploy_model

```python
# 責務
# - Vertex AI Endpoint にモデルをデプロイ
# - BigQuery の Champion レコードを更新
# - Discord に成功通知を送信

# 入力
model_gcs_path: str   # Step3 の出力
endpoint_id: str

# デプロイ戦略
# トラフィック 100% 切り替え（Phase 1）
# ※ Phase 2 で 10% → 100% の段階的切り替えに拡張予定

# 出力
deployed_model_id: str
endpoint_url: str
```

## 7. 品質ゲートの設計思想

### なぜ単純な精度比較ではいけないか

```text
問題: モデルのランダム性によるブレ

RandomForest は学習のたびに微妙に結果が変わる
  └── random_state を固定しても環境差異で変わりうる

素朴な実装（NG）
  └── 「新モデルの RMSE < Champion の RMSE ならデプロイ」
       └── ブレの範囲内の差異で誤判定する

正しい実装
  └── 「新モデルが Champion より有意に良い場合のみデプロイ」
       └── 例: 1% 以上の改善を条件とする
```

### 閾値の外部化

```text
品質ゲートの閾値はハードコードしない
  └── Pipeline 実行パラメータとして外部化
       └── 実験・チューニングが容易になる
       └── 業務環境への応用時に再利用しやすい
```

## 10. 学習ポイント整理

### Vertex AI Pipeline を使うことで理解できる概念

| 概念 | Cloud Run Job との違い | 業務への応用 |
|---|---|---|
| DAG（有向非巡回グラフ） | スクリプトは逐次実行のみ | Pipeline は依存関係で並列化可能 |
| コンポーネント分離 | 1スクリプト（main.py）に全処理 | Step 単位で再実行・差し替え可能 |
| アーティファクト管理 | 手動で GCS パスを管理 | Pipeline が自動でバージョン管理 |
| 品質ゲート | なし（学習したら即保存） | Step として明示的に設計 |
| メタデータ記録 | MLflow で手動 | Vertex AI Experiments と統合 |
| 既存コード再利用 | そのまま動作 | ロジックを Pipeline コンポーネントに変換 |

### Champion/Challenger を実装することで理解できること

```text
1. ベースライン（Champion）の重要性
   └── 比較対象がないと改善を測れない

2. 有意差の定義
   └── 単純比較では不十分
   └── ブレを考慮した閾値設計が必要

3. デプロイ判断の自動化
   └── 人間の勘ではなく指標で判断する

4. 将来の Vertex AI Endpoint トラフィック分割への理解
   └── Phase 2: 10% → 100% 段階的切り替えの土台
```
